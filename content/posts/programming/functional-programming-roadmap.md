이 블로그의 함수형 프로그래밍 글들은 하나의 흐름으로 이어집니다. 에릭 노먼드의 **Grokking Simplicity**에서 출발해, 실용적인 설계 원칙들을 Rust 예제로 하나씩 쌓아갑니다. 핵심 시리즈를 마치면 자율주행·로봇·GPU 도메인에 같은 원칙을 적용하는 응용 시리즈로 이어집니다.

처음 읽는다면 순서대로 따라가는 것을 권장합니다. 특정 주제만 필요하다면 아래 설명을 보고 바로 해당 글로 이동해도 됩니다.

## 읽는 순서

```
[입문]
  모나드로 배우는 함수형 에러 처리
        │
        ▼
[1단계: 코드를 분류하기]
  액션, 계산, 데이터 구분
        │
        ▼
[2단계: 메커니즘 이해하기]
  일급 함수와 클로저
        │
        ▼
[3단계: 함수 연결하기]
  함수 컴포지션 + 모나드
        │
        ▼
[4단계: 구조 잡기]
  계층형 설계
        │
        ▼
[5단계: 시간 다루기]
  타임라인 다이어그램
        │
        ▼
[응용 — 선행 글이 적은 순서로]
  ├──▶ 불변 데이터와 구조적 공유     (← 1번)
  ├──▶ 함수형 DI                     (← 1번 + 2번)
  ├──▶ Iterator 트레잇 직접 구현하기 (← 3번)
  ├──▶ 함수형으로 상태 기계 만들기   (← 1번 + 4번)
  ├──▶ async/await와 타임라인        (← 5번)
  ├──▶ 온어니언 아키텍처             (← 1번 + 4번 + 함수형 DI)
  ├──▶ 파서 컴비네이터               (← 3번 + 모나드)
  └──▶ 중복 제거와 추상화의 기준     (← 4번 + 온어니언)

[자율주행·로봇·GPU 응용 — 응용 단계 이후 권장]
  자율주행·로봇 (순서대로 읽기 권장)
  ├──▶ 자율주행 센서 파이프라인          (← 액션/계산/데이터)
  ├──▶ 함수형 PID 제어기                 (← 불변 데이터 + 액션/계산/데이터)
  ├──▶ 함수형 역기구학                   (← 액션/계산/데이터 + 시뮬레이터·실기기 공유)
  ├──▶ 로봇 경로 계획과 불변 데이터     (← 불변 데이터 + A*)
  ├──▶ ROS2 콜백을 함수형으로            (← 액션/계산/데이터 + 함수형 DI)
  ├──▶ 함수형 포인트 클라우드 처리       (← 액션/계산/데이터 + 센서 파이프라인)
  ├──▶ 함수형 센서 퓨전                  (← 컴포지션 + 추상화 장벽)
  ├──▶ 자율주행 모드 전이를 타입으로     (← 상태 기계 + 함수형 DI)
  └──▶ 시뮬레이션 회귀 테스트 설계      (← 데이터 + 순수 계산 파이프라인, 통합 마무리)

  GPU (독립, 순서 무관)
  ├──▶ 함수형 셰이더 파이프라인          (← 액션/계산/데이터 + GPU 순수 함수)
  └──▶ 함수형 렌더 그래프               (← 액션/계산/데이터 + 셰이더 파이프라인)
```

## 자율주행·로봇·GPU 응용

핵심 시리즈와 응용 글들을 읽은 뒤, 같은 원칙을 자율주행·로봇·GPU 도메인에 적용한 예제입니다. 이론보다 실제 도메인 문제에서 함수형 설계가 어떤 가치를 가지는지 보여줍니다.

### 자율주행·로봇

선행 글이 적은 순서로 나열했습니다. 순서대로 읽기를 권장합니다.

#### 1. [자율주행 센서 파이프라인: 액션/계산/데이터로 설계하기](/posts/programming/autonomous-sensor-pipeline/)

> LiDAR 포인트 클라우드를 필터링·클러스터링해 제어 명령으로 변환하는 파이프라인을 액션/계산/데이터로 나눕니다.

- **핵심 개념**: 센서 I/O를 액션으로 격리, 알고리즘을 순수 계산으로, 실차·시뮬레이터 동일 코드
- **읽고 나면**: 하드웨어 없이 센서 처리 로직을 단위 테스트할 수 있습니다
- **선행 글**: 1번 (액션/계산/데이터)

#### 2. [함수형 PID 제어기: 상태를 값으로 전달하기](/posts/programming/autonomous-pid-controller/)

> 전역 변수로 관리하던 PID 상태를 불변 구조체로 바꾸면, 같은 입력에 항상 같은 출력이 나오고 로그만으로 임의 시점의 제어 출력을 재현할 수 있습니다.

- **핵심 개념**: `PidState` 불변 값 전달, 와인드업 방지, 다중 제어기 독립 운용, 로그 재현
- **읽고 나면**: PID 게인 튜닝 로직을 하드웨어 없이 단위 테스트하고, 주행 로그로 제어 결과를 재현할 수 있습니다
