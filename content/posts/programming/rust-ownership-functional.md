## 이 글을 읽고 나면

- Rust 소유권 규칙이 왜 존재하는지 함수형 관점에서 이해합니다.
- `move`, `&`, `&mut`, `Clone`이 각각 무엇을 보장하는지 직관적으로 압니다.
- 소유권 에러를 만났을 때 "컴파일러가 무엇을 막고 있는가"를 읽을 수 있습니다.

이 글은 [액션/계산/데이터](/posts/programming/functional-actions-calculations-data/) 시리즈의 마무리입니다.

## 소유권은 왜 어렵게 느껴지나

Rust를 처음 배우면 소유권 규칙이 임의적으로 느껴집니다.

> 왜 값을 함수에 넘기면 쓸 수 없게 되는가?  
> 왜 가변 참조는 동시에 하나만 허용하는가?  
> 왜 이렇게까지 엄격한가?

규칙 자체를 외우면 컴파일은 되지만, 왜 그런지 이해하지 못한 채 컴파일러와 싸우게 됩니다.

함수형 프로그래밍의 액션/계산/데이터 구분을 먼저 이해하면 소유권이 다르게 보입니다. Rust의 타입 시스템은 이 구분을 컴파일 타임에 강제하는 장치입니다.

---

## 데이터: 복사할 수 있는 값

함수형 설계에서 데이터는 안전합니다. 언제 만들어도, 얼마나 복사해도 문제가 없습니다.

Rust에서 `Copy` 트레잇이 이것을 표현합니다.

```rust
let x: i32 = 42;
let y = x;       // 복사됨
println!("{}", x); // x는 여전히 유효
```

`i32`, `f64`, `bool` 같은 타입은 `Copy`입니다. 이 값들은 순수한 데이터입니다. 복사해도 원본이 사라지지 않고, 두 복사본이 서로 영향을 주지 않습니다.

**왜 모든 타입이 `Copy`가 아닌가?**

`String`이나 `Vec`은 힙 메모리를 가리킵니다. 단순 복사를 허용하면 두 변수가 같은 메모리를 가리키게 되고, 둘 다 해제를 시도하면 double-free가 됩니다. 복사본이 원본에 영향을 줍니다. 이것은 데이터가 아니라 공유 상태입니다.

---

## 계산: 입력을 받아 출력을 돌려주는 것

계산은 입력과 출력만 있고 외부 상태를 바꾸지 않습니다. Rust에서 이를 가장 잘 표현하는 것은 **불변 참조(`&T`)**입니다.

```rust
fn total_price(items: &[f64]) -> f64 {
    items.iter().sum()
}

let prices = vec![1000.0, 2000.0, 3000.0];
let total = total_price(&prices);
println!("{}", total);   // 6000
println!("{:?}", prices); // prices는 그대로
```

`&T`는 "읽기만 한다"는 계약입니다. 이 함수는:
- 입력(`items`)을 바꾸지 않습니다
- 외부 상태에 영향을 주지 않습니다
- 같은 입력이면 항상 같은 출력을 냅니다

Rust 컴파일러는 `&T`를 받은 함수가 값을 바꾸려 하면 거부합니다. 계산임을 타입으로 보장합니다.

**불변 참조는 동시에 여러 개 가능합니다.**

```rust
let s = String::from("hello");
let r1 = &s;
let r2 = &s;
println!("{} {}", r1, r2); // 둘 다 유효
```

계산은 병렬로 실행해도 안전하다고 했습니다. 읽기만 하는 참조가 여러 개 있어도 서로 방해하지 않습니다. Rust가 이것을 허용하는 이유입니다.

## 액션: 외부 상태를 바꾸는 것

액션은 외부 세계에 영향을 줍니다. Rust에서 가변 참조(`&mut T`)가 이것을 표현합니다.

```rust
fn apply_discount(price: &mut f64, rate: f64) {
    *price *= 1.0 - rate; // 외부 상태를 바꿈
}

let mut price = 10_000.0;
apply_discount(&mut price, 0.1);
println!("{}", price); // 9000
```

`&mut T`는 "이 값을 바꿀 수 있다"는 계약입니다. 그리고 Rust는 **가변 참조를 동시에 하나만 허용합니다.**

```rust
let mut s = String::from("hello");
let r1 = &mut s;
let r2 = &mut s; // 컴파일 에러!
```

왜일까요?

타임라인 다이어그램 글에서 봤던 문제와 같습니다. 두 곳에서 같은 상태를 동시에 바꾸면 순서에 따라 결과가 달라집니다. Rust는 이 상황 자체를 컴파일 타임에 차단합니다.

액션은 신중하게 다뤄야 한다고 했습니다. 가변 참조가 하나뿐이라는 규칙이 바로 그 신중함을 강제합니다.

---

## 소유권 이동: 액션의 책임을 넘기기

`&`나 `&mut` 없이 값을 넘기면 소유권이 이동합니다.

```rust
fn process(order: Order) {
    // order를 소비함
}

let order = Order { id: 1, item: "pen".into() };
process(order);
// println!("{}", order.id); // 컴파일 에러: order는 이미 이동됨
```

이것은 "이 값에 대한 모든 책임을 넘긴다"는 뜻입니다.

함수형 관점에서 보면, 소유권 이동은 **액션의 책임 이전**입니다. `process`가 `order`를 받는 순간, `order`의 생명주기 관리는 `process`의 책임이 됩니다. 이전 소유자는 더 이상 이 값에 접근할 수 없습니다.

이것이 이상하게 느껴지는 이유는 대부분의 언어가 이 책임을 GC에 맡기기 때문입니다. Rust는 책임을 명시적으로 추적합니다.

**언제 소유권을 이동하는가?**

값을 소비하고 반환하지 않을 때입니다. 계산(읽기만 하는 경우)이라면 `&T`를, 값을 완전히 소비하는 액션이라면 소유권 이동을 씁니다.

```rust
// 계산: 읽기만 함 → &T
fn render_email(order: &Order) -> String { ... }

// 액션: 소비함 → 소유권 이동
fn save_to_db(order: Order) { ... }
```

## `Clone`: 데이터를 명시적으로 복제하기

`Copy`가 아닌 값을 두 곳에서 써야 할 때 `Clone`을 씁니다.

```rust
let original = Order { id: 1, item: "pen".into() };
let copy = original.clone();

render_email(&original); // 계산
save_to_db(copy);        // 액션 (소유권 이동)
```

`Clone`이 명시적인 이유가 있습니다. 복제는 비용이 있고, 복제본은 원본과 독립된 데이터입니다. 암묵적으로 일어나면 어디서 복사가 일어나는지 추적하기 어렵습니다.

함수형 관점에서 `Clone`은 "새로운 데이터를 만든다"는 선언입니다. 공유 상태 없이 각자 독립된 값을 갖습니다.

## 소유권 규칙을 함수형 원칙으로 읽기

지금까지의 내용을 정리하면 이렇습니다.

| Rust 개념 | 함수형 의미 | 보장하는 것 |
|---|---|---|
| `T` (소유권) | 책임 이전 | 해제 시점이 명확함 |
| `&T` (불변 참조) | 계산의 입력 | 읽기만 함, 병렬 안전 |
| `&mut T` (가변 참조) | 액션의 대상 | 동시에 하나만 바꿀 수 있음 |
| `Copy` | 순수한 데이터 | 복사해도 독립적 |
| `Clone` | 명시적 데이터 복제 | 비용을 의도적으로 지불 |

소유권 에러는 대부분 이 원칙 중 하나를 위반했을 때 납니다.

- **"값이 이동됐다"** → 계산(읽기)인데 소유권을 이동시켰다. `&T`로 바꾼다.
- **"가변 참조가 두 개"** → 두 곳에서 동시에 액션을 시도했다. 하나를 `&T`로 바꾸거나 순서를 직렬화한다.
- **"수명이 충분하지 않다"** → 데이터보다 참조가 오래 살려 한다. 소유권을 올리거나 `Clone`한다.

## 소유권 에러를 읽는 연습

다음 코드는 컴파일되지 않습니다. 왜인지 생각해봅니다.

```rust
fn double_spend(order: Order) {
    save_to_db(order);    // 소유권 이동
    send_email(order);    // 에러: order는 이미 이동됨
}
```

함수형으로 읽으면: `save_to_db`가 `order`의 책임을 가져갔는데, `send_email`도 같은 값에 액션을 시도합니다. 같은 주문을 두 번 처리하는 것입니다. Rust가 막는 게 맞습니다.

해결책은 세 가지입니다.

```rust
// 1. 읽기만 한다면 참조로
fn process(order: &Order) {
    save_to_db_ref(order);
    send_email_ref(order);
}

// 2. 순서가 중요하다면 명시적으로
fn process(order: Order) {
    send_email_ref(&order); // 먼저 읽고
    save_to_db(order);      // 마지막에 소비
}

// 3. 독립적인 복사본이 필요하다면
fn process(order: Order) {
    save_to_db(order.clone());
    send_email(order);
}
```

어떤 해결책이 맞는지는 `save_to_db`와 `send_email`이 계산인지 액션인지, 순서가 중요한지에 따라 결정됩니다.

---

## 마치며

Rust 소유권은 처음에는 제약처럼 느껴집니다. 하지만 함수형 관점에서 보면 이것은 제약이 아니라 **설계 원칙을 강제하는 장치**입니다.

- 계산은 `&T`로 받는다 — 읽기만 한다는 것을 타입으로 선언
- 액션은 `&mut T`나 소유권 이동으로 받는다 — 변경의 책임이 명확
- 공유 가변 상태는 컴파일 타임에 차단된다 — 타임라인 충돌을 사전에 방지

함수형 프로그래밍이 "액션을 최소화하고 명시적으로 다뤄라"고 말할 때, Rust는 그것을 선택이 아니라 규칙으로 만듭니다. 컴파일러가 통과시켜준 코드는 이미 많은 함수형 원칙을 지키고 있습니다.
